算法解谜游戏：“深渊逃生” (Grid Escape) 完整开发文档
1. 项目概述

“深渊逃生” (Grid Escape) 是一款将经典寻路算法与 Roguelike 解谜元素结合的网页游戏。玩家被困在充满“战争迷雾”与“泥沼”的地下深渊中，需要通过编写或选择正确的路径搜索算法（如 BFS, Dijkstra, A*）来寻找最优逃生路线，同时管理有限的“体力值”。

核心目标：

    提供直观、具象化的算法执行过程展示（Open Set / Closed Set 扫描动画）。

    将枯燥的算法学习转化为具有挑战性的资源管理与路径规划游戏。

2. 技术栈选型

    前端框架: React 18 + Vite (极致的启动与热更新速度，非常适合 Ubuntu 本地开发环境)

    开发语言: TypeScript (强类型约束，极大降低图论算法中的边界错误，利于 AI 理解上下文)

    状态管理: Zustand (轻量级，避免 React Context 在高频网格更新时的渲染性能衰减)

    样式方案: Tailwind CSS (原子化 CSS，快速构建深色系暗黑风格 UI)

    动画引擎: Framer Motion (用于处理节点扫描的丝滑扩散效果和角色移动路径)

4. 核心数据结构与状态设计

系统设计的核心在于 Node 接口的设计。每个网格单元不仅是视觉元素，也是图算法中的一个顶点。
src/algorithms/index.ts (类型契约)
TypeScript

export type NodeType = 'empty' | 'wall' | 'weight' | 'start' | 'end';

export interface GridNode {
  row: number;
  col: number;
  type: NodeType;
  isVisited: boolean;      // 算法是否扫描过 (用于渲染 Open/Closed Set)
  isPath: boolean;         // 是否属于最终计算出的逃生路径
  distance: number;        // 起点到该节点的距离 (Dijkstra/A* 使用)
  heuristic?: number;      // 启发式预估距离 $h(n)$ (A* 使用)
  previousNode: GridNode | null; // 记录父节点，用于算法结束后回溯路径
}

// 算法函数的标准输出
export interface AlgorithmResult {
  visitedNodesInOrder: GridNode[]; // 记录扫描顺序，用于逐帧播放动画
  pathNodesInOrder: GridNode[];    // 最终的最短路径
  success: boolean;                // 是否成功找到终点
}

5. 游戏化核心机制

    战争迷雾 (Fog of War)：

        初始状态下，只有 startPos 周围半径为 3 的区域可见。

        随着算法的扫描 (visitedNodesInOrder)，动态更新迷雾遮罩层，揭开地图。

    体力/能量系统：

        基础移动消耗：跨越普通格子消耗 10 点能量。

        泥沼地形消耗：跨越 isWeight 格子消耗 30 点能量。

        如果在能量耗尽前 pathNodesInOrder 未抵达终点，则触发 Game Over 弹窗。

    启发式算法优势体现：

        A* 算法由于引入了评估函数 f(n)=g(n)+h(n)，其扫描的节点数远少于 Dijkstra。在游戏中，扫描多余节点将扣除“时间分数”，鼓励玩家在复杂关卡优先选择 A* 算法。

6. Cursor 辅助开发步骤路线图

在 Ubuntu 终端初始化项目后（npm create vite@latest grid-escape -- --template react-ts），按以下顺序在 Cursor 中使用 Composer 或 Chat 进行迭代：

    Phase 1: 基础设施建设

        提示词方向：根据 GridNode 接口，使用 Zustand 初始化一个 20x20 的网格状态。编写 GridContainer 和 Node 组件渲染网格。

        目标：屏幕上出现网格，可以通过鼠标点击设置起点和终点。

    Phase 2: 交互与路障绘制

        提示词方向：在 GridContainer 中添加 onMouseDown, onMouseEnter, onMouseUp 事件处理，实现按住鼠标拖拽可以连续绘制“墙壁”(wall) 的功能。

    Phase 3: 引入算法引擎

        提示词方向：在 src/algorithms/bfs.ts 中实现广度优先搜索。函数签名需严格遵守接收 grid, startNode, endNode，返回 visitedNodesInOrder。处理墙壁碰撞逻辑。

    Phase 4: 动画调度器 (Visualizer)

        提示词方向：编写 useAlgorithm Hook。读取算法返回的 visitedNodesInOrder 数组，使用 setTimeout 循环，每隔 15ms 依次更新对应 Node 的 isVisited 状态为 true。扫描完成后，再依次绘制最终 path。

    Phase 5: 游戏逻辑封装

        提示词方向：引入体力值和迷雾层。使用 Framer Motion 为 isVisited 和 isPath 状态变化的单元格添加缩放和变色动画，增强视觉冲击力。
